# Spike B — Bengali cluster reordering correctness & idempotency

**Question.** Can we convert Bijoy/SutonnyMJ to Unicode with *correct cluster
reordering* (pre-base vowels ি/ে/ৈ moved after their consonant; reph র্ moved to
the cluster start; o-kar/au-kar composed), in a way that is **idempotent**
(running it twice does not corrupt text)?

**Why it's a killer.** This is the single hardest part of the conversion, and it
is exactly where the prior engine failed: its reorder step re-fired on its own
output, so converting twice produced garbage (e.g. কিছু → কছিু).

## Verdict: 🟢 GREEN

A clean-room reference converter
([`tools/corpus/reference-converter.mjs`](../../../tools/corpus/reference-converter.mjs))
— authored independently from the cited mapping research, **not** from any
Mukti-lineage code — scores, against the frozen corpus:

| Set | Cases | Char accuracy | Word accuracy | Idempotency | No-op on Unicode | Fuzz (2000) |
|---|---|---|---|---|---|---|
| `visible` | 92 | **100.00%** | **100.00%** | ✅ pass | ✅ pass | ✅ no crashes / all NFC |
| `heldout` | 15 | **100.00%** | **100.00%** | ✅ pass | ✅ pass | ✅ no crashes / all NFC |

Reproduce:
```
node tools/corpus/harness.mjs --dir corpus/visible  --converter tools/corpus/reference-converter.mjs
node tools/corpus/harness.mjs --dir corpus/heldout   --converter tools/corpus/reference-converter.mjs
```

### What makes this result trustworthy (not circular)

The corpus `expected` values were generated by **one** independent oracle
(rabiulislam-xyz, a different lineage from Mukti). The spike converter is a
**second**, independently-written implementation. Two independent
implementations agreeing on all 107 cases — including a sealed held-out set —
is strong evidence the reordering rules are correct, not a coincidence of shared
code.

### The idempotency finding (the important part)

The first version of the spike converter scored **100% accuracy but FAILED
idempotency**: feeding already-Unicode text back in, a blind regex reorder
wrongly swapped a pre-base vowel with the *following* consonant (দেশ → দশে). This
is the same bug class that sank the prior engine.

**The fix — a design rule we now carry into the engine:** a conversion only runs
on text that actually contains Bijoy *source* glyphs. The output of a conversion
contains none, so any second pass is a guaranteed no-op. With this guard,
`convert(convert(x)) === convert(x)` and `convert(unicode) === unicode` both hold
across the whole corpus. The production engine must enforce this same rule (and
the harness tests it on every case).

### Scope / honesty notes

- The reference converter's glyph map covers the corpus glyph set; **completeness
  of the full SutonnyMJ glyph map is mundane Phase-4 data work**, not a
  reordering risk. The *reorder logic* is general and was validated on held-out
  words it was not written against.
- This is a spike artifact, not the production engine. The production engine
  (`src/engine`, data-driven from `data/bijoy-sutonnymj.json`) is now built and is
  the authority: it passes the **full** corpus (visible + held-out) at 100% via
  `npm run corpus:gate`. As the corpus grew (Phase 3/6), the reference converter's
  intentionally-small clean-room map no longer covers every case — that is
  expected; it served its purpose (proving the reorder algorithm), and the gate
  now runs against the production engine, not this artifact.
